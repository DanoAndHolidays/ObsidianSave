## 这个虚拟滚动的实现原理 —— 完整拆解

我会把它分成 **6 个核心环节** 来讲，每个环节解决一个问题。

---

### 一、核心问题：为什么需要虚拟滚动？

假设你有 **10 万个树节点**。如果全部渲染成 DOM，浏览器会直接卡死。虚拟滚动的思路是：

> **"我只渲染用户当前看得见的那几十行，看不见的用空白占位，欺骗用户的眼睛。"**

这个组件通过 `position: absolute` + `top` 把每一行精确定位，用外层包裹的 `<div>` 的高度来撑开滚动条。

---

### 二、数据结构设计（Line 10-16, 39-50）

```ts
interface FlatItem {
  id: string       // 节点 ID
  level: number    // 树层级 (根=0, 子=1, 孙=2...)
}
```

核心思想是**把树拍平成列表**。`flattenTree` 函数做了这件事：

```ts
function flattenTree(nodeMap, ids, level, expandedIds, out) {
  for (const id of ids) {
    out.push({ id, level })
    // 如果当前节点已展开 且 有子节点，递归展平子节点
    if (expandedIds.has(id) && node.childrenIds.length > 0) {
      flattenTree(nodeMap, node.childrenIds, level + 1, expandedIds, out)
    }
  }
}
```

**只看已展开的节点**，按树的深度优先遍历顺序，产出一个一维数组 `flatList`。

举例：树是 `A → [B, C]`，B 展开后是 `[D, E]`，则 `flatList` = `[A, B, D, E, C]`。

这个 `flatList` 就是虚拟滚动的"总数据源"，`flatList.length` 就是"总行数"。

---

### 三、核心数学：哪些行需要渲染？（Line 77-85, 92-95）

这是整个组件的心脏。看图：

```
┌─────────────────────────────┐
│  overscan 上缓冲区 (看不见)  │  ← renderStart
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │  ← viewportStart
│                             │
│    视口区域 (用户看得见)      │  ← 只需要渲染这部分!
│                             │
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │  ← viewportEnd
│  overscan 下缓冲区 (看不见)  │  ← renderEnd
└─────────────────────────────┘
```

计算过程分两步：

**第一步 — 计算精确视口范围：**

```ts
const viewportStart = Math.floor(scrollTop / ITEM_HEIGHT)
const viewportEnd   = Math.ceil((scrollTop + containerHeight) / ITEM_HEIGHT)
```

这是纯数学：滚动偏移量 ÷ 行高 = 第一条可见行的索引。

**第二步 — 加上 overscan 缓冲区：**

```ts
const renderStart = Math.max(0, viewportStart - OVERSCAN)     // 上扩 15 行
const renderEnd   = Math.min(flatList.length, viewportEnd + OVERSCAN)  // 下扩 15 行
```

**为什么需要 overscan？** 用户快速滚动时，如果只渲染视口内的行，会出现短暂的"白屏"——新行还没渲染出来。overscan 在视口上下各多渲染 15 行作为缓冲，等用户滚到那里时 DOM 已经就位了。

`OVERSCAN = 15` 意味着实际渲染行数 = 视口行数 + 30（上下各 15）。

---

### 四、渲染：绝对定位 + 动态计算 top（Line 148-220）

```tsx
{visibleItems.map(({ flatItem, index }) => (
  <div
    style={{
      position: 'absolute',      // 绝对定位!
      top: index * ITEM_HEIGHT,  // 用 index 乘以行高算出位置
      height: ITEM_HEIGHT,
    }}
  >
    {node.label}
  </div>
))}
```

**关键点：** 每行的 `top` 不依赖于它前面有没有渲染 DOM，而是直接用 `index * 36px` 算出来的。

这意味着：
- 第 0 行（无论是否渲染）理论上在 top: 0
- 第 5000 行理论上在 top: 180000px
- 我们只渲染 4955~5025 这几行，它们各自 `top: 4955*36` 到 `top: 5025*36`

**外层容器**通过 `height: totalHeight` 撑开滚动条：

```ts
const totalHeight = flatList.length * ITEM_HEIGHT  // 10万行 = 360万px
```

浏览器认为有 360 万像素的内容，所以滚动条长度和位置都是对的。但实际上 DOM 里只有几十个 `<div>`。

---

### 五、ResizeObserver 动态感知容器高度（Line 53-63）

```ts
useEffect(() => {
  const ro = new ResizeObserver((entries) => {
    const h = entries[0]?.contentRect.height
    if (h && h > 0) setContainerHeight(h)
  })
  ro.observe(el)
  return () => ro.disconnect()
}, [])
```

当浏览器窗口大小变化时，容器高度会变。`ResizeObserver` 实时捕获新高度，触发重新计算视口范围，保证渲染的行数始终适配当前窗口。

---

### 六、越界修正（Line 73-81）—— 防止展开/收起时滚动条"溢出"

```ts
useEffect(() => {
  const maxScroll = Math.max(0, totalHeight - containerHeight)
  if (container.scrollTop > maxScroll) {
    container.scrollTop = maxScroll
  }
}, [flatList.length, containerHeight, totalHeight])
```

场景：用户展开了一个大节点，`flatList` 变长，`totalHeight` 增大，但 `scrollTop` 没变。如果收缩一个节点，`totalHeight` 变小，可能导致 `scrollTop > 总高度`，滚动条"越界"。这个 effect 会在必要时把 `scrollTop` 修正到合法范围。

```ts
ZONE.ABOVE    = 'overscan-above'    // 上方缓冲区：暖橙色 #fff7e6
ZONE.VIEWPORT = 'viewport'          // 可见视口：白色   #ffffff
ZONE.BELOW    = 'overscan-below'    // 下方缓冲区：浅蓝色 #e6f7ff
```

每行根据自己落在哪个 zone 来变色。还有红色/橙色的虚线标记边界，便于观察虚拟滚动的行为。

## 总结：一句话讲清楚

> **"把树拍平成带层级的列表 → 根据 scrollTop 算出可见行范围 → 只在视口 + 上下 15 行缓冲区里渲染 DOM → 用绝对定位把每行放在正确位置 → 外层容器用总行数 × 行高撑开滚动条。滚动条是真的，DOM 是假的。"**

这就是虚拟滚动的本质：**用数学骗过眼睛，用绝对定位骗过浏览器。**

好问题！这涉及到 **两套"可见"概念**，非常容易混淆。让我讲清楚：

## 区分两种"可见"

### 第一套：树的展开/收起 → 决定节点"是否在 flatList 里"

这是你问的核心。看 [flattenTree](file:///g:/Save/Grogramming/Vue3/yike/src-react/views/TreeDemo/TreeVirtualList.tsx#L18-L28)：

```ts
function flattenTree(nodeMap, ids, level, expandedIds, out) {
  for (const id of ids) {
    out.push({ id, level })                      // ① 当前节点一定加入 flatList
    if (expandedIds.has(id) && node.childrenIds.length > 0) {
      flattenTree(nodeMap, node.childrenIds, level + 1, expandedIds, out)  // ② 只有展开了才递归子节点
    }
  }
}
```

**逻辑非常直接：**

- **每个父节点自己**永远在 `flatList` 里（第 ① 步）
- 但它的**子节点**是否进入 `flatList`，取决于父节点是否在 `expandedIds` 里（第 ② 步的判断）

举个具体例子，假设树结构是：

```
A
├── B
│   ├── D
│   └── E
└── C
    └── F
```

#### 场景 1：全部收起（`expandedIds = {}`）

`flatList` = `[A, B, C]`

D、E、F 根本不在列表里——它们的父节点没展开，所以不会被遍历到。

#### 场景 2：展开 A，但 B 和 C 收起（`expandedIds = {A}`）

`flatList` = `[A, B, C]`

A 展开了 → 遍历了 B 和 C（它们加入列表）。但 B 没展开 → D、E 不加入。

#### 场景 3：展开 A 和 B（`expandedIds = {A, B}`）

`flatList` = `[A, B, D, E, C]`

A 展开 → B 加入 → B 展开 → D、E 加入。C 加入但不展开。

#### 场景 4：展开 A 和 C（`expandedIds = {A, C}`）

`flatList` = `[A, B, C, F]`

A 展开 → B、C 加入。C 展开 → F 加入。B 没展开，D、E 不加入。

---

### 第二套：虚拟滚动的视口裁剪 → 决定节点"是否渲染 DOM"

这个我们上一轮讲过了。**在 `flatList` 里的所有节点**，还会进一步被虚拟滚动筛选：

```
flatList 有 50,000 个节点
        │
        ▼ 虚拟滚动裁剪
只渲染第 500~560 行（视口 + overscan 缓冲区）
```

## 关键总结

| 概念 | 谁控制 | 决定的含义 | 实现位置 |
|------|--------|-----------|---------|
| **树可见性** | `expandedIds` (Set) | 子节点是否出现在列表中 | `flattenTree()` |
| **DOM 可见性** | `scrollTop` × `containerHeight` | 是否创建 DOM 元素 | `renderStart/renderEnd` |

**一句话：`expandedIds` 决定"这个节点有没有资格出现在列表里"，虚拟滚动决定"有资格的节点里，哪些真正画到屏幕上"。**

所以你看 [Line 95-105](file:///g:/Save/Grogramming/Vue3/yike/src-react/views/TreeDemo/TreeVirtualList.tsx#L92-L108) 的 `visibleItems` —— 它遍历的是 `flatList[renderStart..renderEnd]`，能出现在这里的，首先是**已经被展开链路上的节点**（通过了 `flattenTree` 的筛选），其次才经过虚拟滚动的位置裁剪。

非常好的问题！答案是：**不会，每次滚动完全不会重新执行 `flattenTree`。**

让我用代码证明这一点。

`flattenTree` 被包在 `useMemo` 里，依赖项是 `[nodeMap, rootIds, expandedIds]`。

## 滚动时到底发生了什么事？

当用户滚动时，调用链是这样的：

```
用户滚动
  → handleScroll 触发        (Line 113-115)
  → setScrollTop(新值)        ← 只改了一个 state
  → 组件重新渲染
  → viewportStart/End 重新计算  (Line 79-83) — 纯算术，O(1)
  → renderStart/End 重新计算    (Line 86-87) — 纯算术，O(1)
  → flatList 的 useMemo 检查依赖：nodeMap 没变、rootIds 没变、expandedIds 没变
  → ❌ 跳过 flatList 计算！直接用缓存！
  → visibleItems 重新计算       (只是对 flatList 做切片 for 循环，O(30~60))
  → 重新渲染那几十个 DOM
```

---

## `flattenTree` 什么时候才真正跑？

**只在三种情况下：**

| 触发条件 | 用户操作 |
|---------|---------|
| `nodeMap` 变了 | 数据源换了（几乎不发生） |
| `rootIds` 变了 | 数据源换了（几乎不发生） |
| `expandedIds` 变了 | 点击展开/收起箭头 |

也就是说，`flattenTree` 只在用户**点击展开/收起**时才执行，而且执行一次后就缓存，后续滚动完全不走。

## 一句话总结

> **`flattenTree` 的结果被 `useMemo` 缓存了，依赖项里没有 `scrollTop`。滚动 1000 次，`flattenTree` 执行 0 次。只有点展开/收起才重新计算。**

这就是 React 虚拟滚动的标准优化手法：**把昂贵的"树拍平"和廉价的"切片取范围"分离**，前者用 `useMemo` 缓存，后者每次滚动时 O(1) 计算。

好问题！这其实涉及两个层面：**展开时的视觉效果** 和 **展开后的时序**。

## 第一问：为什么展开时画面不会跳？

答案其实很自然——**展开的那个节点本身，位置不动。**

### 关键原因：父节点永远在列表里

回到 `flattenTree` 的逻辑：

```ts
function flattenTree(nodeMap, ids, level, expandedIds, out) {
  for (const id of ids) {
    out.push({ id, level })    // ← 父节点自己先加入
    if (expandedIds.has(id) && node.childrenIds.length > 0) {
      flattenTree(...)          // ← 子节点跟在后面加入
    }
  }
}
```

**父节点本来就在 `flatList` 里**（无论展开还是收起），展开只是把子节点"插入到它后面"。

### 图例说明

```
展开前：                           展开后：
┌── A ── (index 0)                ┌── A ── (index 0)
├── B ── (index 1)  ◁ 点这里       ├── B ── (index 1)  ← 位置没变！
├── C ── (index 2)                │   D ── (index 2)  ← 新插入
└── ...                           │   E ── (index 3)  ← 新插入
                                  ├── C ── (index 4)  ← 被推下去了
                                  └── ...
```

用户点击 B 的时候，B 在屏幕上的位置是 `index * 36px`。展开后 B 的 index 还是 1，所以它的 `top: 36px` 完全不变。**因为绝对定位用的是 index，不是"前面有多少个可见 DOM"，所以 B 纹丝不动。**

变化的只是 B 下面的内容被"撑开了"——C 从 index=2 变成了 index=4，往下移了 72px。但这是**自然的、符合预期的向下平移**，不是"跳动"。

## 但是，这里有一个精巧的边界处理

看 [Line 74-81](file:///g:/Save/Grogramming/Vue3/yike/src-react/views/TreeDemo/TreeVirtualList.tsx#L73-L81) 的越界修正：

```ts
useEffect(() => {
  const maxScroll = Math.max(0, totalHeight - containerHeight)
  if (container.scrollTop > maxScroll) {
    container.scrollTop = maxScroll
  }
}, [flatList.length, containerHeight, totalHeight])
```

这个 effect 专门处理**收起节点**时的特殊情况：

```
场景：用户在列表底部，展开了一个大节点
  → 往下滚了很远，看到了很多内容
  → 然后收起一个上层节点
  → flatList 变短，totalHeight 骤降
  → scrollTop 可能 > 新的 totalHeight - containerHeight
  → 如果不修正，滚动条会"悬空"
  → 这个 effect 自动把 scrollTop 钳制到合法范围
```

而对于**展开**，不需要特殊处理——`totalHeight` 变大了，原来的 `scrollTop` 还在合法范围内，画面自然向下扩展。

## 场景一：虚拟滚动 + 分页（懒加载树）

> 数据量极大（百万级），不可能一次全拉到前端。

### 核心矛盾

当前架构的前提是：**`nodeMap` 里已经有了整棵树**，`flattenTree` 才能递归。如果数据是分页从后端拿的，你根本不知道"B 下面有没有孙子节点"，也就没法预先算出 `flatList` 的总长度。

### 解决方案：展开时才加载子节点（Lazy Tree）

```
┌──────────────────────────────────────────────────┐
│                   初始状态                         │
│  后端返回第 1 页: 100 个根节点（只有 label + id）    │
│  flatList = [集团-0001, 集团-0002, ...集团-0100]   │
│  totalHeight = 100 × 36 = 3600                    │
│                                                   │
│  虚拟滚动照常工作，只渲染视口内的根节点               │
└──────────────────────────────────────────────────┘
                        │
                        ▼ 用户点击「集团-0001」的展开箭头
┌──────────────────────────────────────────────────┐
│  后端请求: GET /api/tree/children?parentId=node_0   │
│  返回: { children: [事业部-0001, 事业部-0002, ...] } │
│                                                   │
│  前端: nodeMap.set(子节点们)                        │
│        node_0.childrenIds = [...]                 │
│        expandedIds.add('node_0')                  │
│        重新 flattenTree → flatList 多了这 3 个子节点  │
│                                                   │
│  UI: 集团-0001 位置不变，子节点出现在它下方            │
└──────────────────────────────────────────────────┘
```

### 关键设计点

**1. 节点需要标记"是否已加载子节点"：**

```ts
interface TreeNode {
  id: string
  label: string
  hasChildren: boolean       // 后端告知"有没有孩子"
  childrenLoaded: boolean    // 前端已请求过子节点?
  childrenIds: string[]      // 已加载的子节点 ID
  level: number
}
```

- `hasChildren: true` → 显示 ▶ 箭头，可以点
- `childrenLoaded: false` → 点箭头时发请求
- `childrenLoaded: true` → 点箭头时直接切换展开/收起（纯本地）

**2. `totalHeight` 不再是最终高度：**

当前代码的 `totalHeight = flatList.length * ITEM_HEIGHT` 依然成立，但它是"已加载数据的行数 × 行高"。随着不断展开，`flatList` 越来越长，滚动条也越来越长，**用户体验和"一次性全量"完全一致**——区别只是没加载的节点你永远碰不到。

**3. 虚拟滚动逻辑完全不用改：**

```ts
// 这段逻辑一行都不用动
const viewportStart = Math.floor(scrollTop / ITEM_HEIGHT)
const renderStart = Math.max(0, viewportStart - OVERSCAN)
// flatList 多了新节点 → 自动生效
```

**4. 需要额外处理的一种情况——节点在视口范围内但子节点还在加载中：**

可以在箭头旁边放一个 loading 动画：
```tsx
{node.hasChildren && !node.childrenLoaded && isExpanded && (
  <Spin size="small" />  // 正在请求子节点...
)}
```

## 场景二：纯分页（传统翻页）

> 不要虚拟滚动，用传统的"上一页 / 下一页"。

### 核心区别

| | 虚拟滚动 + 分页 | 纯分页 |
|---|---|---|
| 用户交互 | 滚轮 + 展开箭头 | 点"上一页/下一页" |
| DOM 节点数 | 始终 ~50 个 | 每页固定 N 个（如 100 个） |
| 滚动条 | 真实的，长度 = 总数据量 | 无所谓，容器不需要 overflow |
| 树结构处理 | 自然，展开即可 | 需要用页码代替滚动 |

### 设计

纯分页对**树**其实非常不友好，因为树的"第 N 页"没有统一标准。通常你只能对**某一层**做分页。

**最简单的方案：只对扁平视图分页**

```ts
// 把当前展开状态下的 flatList 分页
const PAGE_SIZE = 100
const totalPages = Math.ceil(flatList.length / PAGE_SIZE)
const pageData = flatList.slice(
  (currentPage - 1) * PAGE_SIZE,
  currentPage * PAGE_SIZE
)
```

但这个方案有致命问题：**展开一个节点后，flatList 长度变了，当前页码可能越界**。

**更好的方案：后端分页 + 前端树结构**

```
API: GET /api/tree/roots?page=1&pageSize=50
返回:
{
  items: [集团-0001, 集团-0002, ...],   // 第 1 页的根节点
  total: 100,                           // 总共 100 个根节点
  page: 1,
  pageSize: 50,
  totalPages: 2
}

API: GET /api/tree/children?parentId=node_0&page=1&pageSize=20
返回:
{
  items: [事业部-0001, ...],            // node_0 的第 1 页子节点
  total: 85,                            // node_0 总共 85 个子节点
  page: 1,
  totalPages: 5
}
```

每个展开的节点都有自己的分页状态：

```ts
// 前端状态
const [expandedIds, setExpandedIds] = useState<Set<string>>(new Set())
const [childPages, setChildPages] = useState<Map<string, {
  items: TreeNode[]
  total: number
  currentPage: number
}>>(new Map())
```

渲染时：

```
┌── 集团-0001 ───────────────────────┐
│  ├── 事业部-0001                    │
│  ├── 事业部-0002                    │
│  ├── ...                           │
│  ├── 事业部-0020                    │
│  │  ◀ 第 1/5 页  下一页 ▶          │ ← 分页器
└────────────────────────────────────┘
```

每个展开的节点内部嵌入一个微型分页器，点击换页只替换该节点下的子节点列表。

### 对比总结

```
┌─────────────────┬──────────────────────┬─────────────────────┐
│                 │ 虚拟滚动 + 懒加载      │ 纯分页               │
├─────────────────┼──────────────────────┼─────────────────────┤
│ 用户体验         │ ★★★★★ 丝滑滚动       │ ★★★ 频繁点击翻页      │
│ 实现复杂度       │ ★★★★ 较高            │ ★★ 较低              │
│ 适合场景         │ 浏览/探索大量节点     │ 查找特定页           │
│ 搜索/跳转支持    │ 需要额外做            │ 天然支持页码跳转      │
│ 树结构友好度     │ ★★★★★ 展开即所见      │ ★★ 分页破坏层级感    │
│ 后端接口         │ 按 parentId 查子节点  │ 同上 + 需返回 total   │
│ scrollTop 状态   │ 需要保持              │ 不需要               │
└─────────────────┴──────────────────────┴─────────────────────┘
```

**我的建议：** 树形结构数据天然适合虚拟滚动 + 懒加载。纯分页更适合扁平列表（如表格），给树用分页会破坏层级导航的体验——用户想看"集团-0001 下面第 3 页的子节点"本身就很不直觉。

这个问题问得很关键！但其实**当前实现已经天然支持这种嵌套懒加载**，不需要额外改造。让我画个流程说清楚。

## 核心原理：每层独立懒加载

关键在于 [index.tsx 的 handleToggle](file:///g:/Save/Grogramming/Vue3/yike/src-react/views/TreeDemo/index.tsx#L25-L53)：

```ts
const handleToggle = useCallback((id: string) => {
  const node = treeData.nodeMap.get(id)
  if (!node) return

  if (node.hasChildren && !node.childrenLoaded) {
    // ← 任何层级、任何节点，只要子节点没加载就发请求
    setLoadingIds((prev) => new Set(prev).add(id))
    loadChildren(id).then(() => {        // ← 同一个 loadChildren，对 id=node_0 或 id=node_100 都管用
      setLoadingIds((prev) => { ... })
      setDataVersion((v) => v + 1)
      setExpandedIds((prev) => new Set(prev).add(id))
    })
  } else {
    // 子节点已加载 → 纯本地切换展开/收起
    setExpandedIds(...)
  }
}, [treeData, loadChildren])
```

`loadChildren(id)` 不关心 `id` 是哪个层级——它在 [mockData.ts](file:///g:/Save/Grogramming/Vue3/yike/src-react/views/TreeDemo/mockData.ts#L166-L246) 里根据 `parent.level + 1` 自动生成对应层级的子节点：

```ts
const childLevel = parent.level + 1      // 根节点 level=0 → 生成 level=1
                                         // 事业部 level=1 → 生成 level=2
                                         // 部门 level=2     → 生成 level=3
```

你提到"根节点下的 1 级节点是分页的"——如果某个节点有 100 个子节点，不想一次全拉，可以这样改造 `loadChildren`：

```ts
// mockData.ts — 分页版 loadChildren
async function loadChildren(parentId: string, page = 1, pageSize = 20) {
  // 返回 { items: TreeNode[], total: 100, page, pageSize }
}
```

然后在 `TreeNode` 加一个字段追踪当前加载到第几页了：

```ts
interface TreeNode {
  // ...existing fields
  hasNextPage: boolean        // 是否还有下一页子节点
  childrenPage: number        // 当前加载到第几页
}
```

每个节点下方渲染一个「加载更多」按钮：

```
┌── 事业部-0001 ▼ ──── badge: 100
│   ├── 部门-0001
│   ├── ...          (第 1 页，前 20 个)
│   ├── 部门-0020
│   └── [加载更多...]  ← 点击加载第 2 页
```

这其实是一种**混合策略**：横向用「懒加载展开」管层级，纵向用「分页加载更多」管子节点数量爆炸。但当前 10 万节点的 mock 场景里每个父节点最多 4 个子节点，分页没必要，所以当前实现没做。
